> **對應 SDLC 階段**：需求分析 (Requirements Analysis)

## 1. 專案概述 (Project Overview)

### 1.1 背景與動機
在現代社會中，購買運動彩券（運彩）已成為許多人在休閒娛樂之餘，期盼能獲得額外收益或「發大財」的合法管道。然而，多數購買者在缺乏數據支持的情況下，往往憑直覺或明牌盲目投注，導致「贏得少、輸得多」，甚至越輸越多。

為了從根本上解決投注缺乏科學依據、勝率低下的問題，本專案旨在設計一個具備科學統計與分析的**運彩分析系統**。透過歷史中獎號碼的統計與數學機率分析，為使用者提供理性的數據支持，提高中獎機率；同時，藉由個人投注記帳功能，讓使用者明晰自己的盈虧狀況，達到理性投資、防範過度投機的目的。

### 1.2 目標用戶
1. **運彩/彩券常客 (如：家長、長輩)**：希望透過更科學、數學化的分析方法來取代純直覺投注。
2. **新手投注者**：突然想碰碰運氣但希望能得到數據建議、提升勝率的人。
3. **想理性控制花費的用戶**：希望透過詳細記帳，了解自己到底在彩券上花費與虧損了多少，避免過度沉迷。

### 1.3 核心價值主張
* **數據透明化**：將過往複雜的彩券中獎歷史整理為清晰的統計數據。
* **預測輔助**：利用科學統計演算法，產出機率較高的數字組合，拒絕盲目猜測。
* **理性投資**：透過「花了多少錢」的盈虧記帳功能，協助使用者了解投資回報率，甚至幫助沉迷者因看清虧損事實而適時收手。

## 2. 功能需求 (Functional Requirements)

本系統規劃了五大核心功能（F-01 至 F-05）。以下使用使用者故事（User Story）格式進行詳細描述：

### F-01：統計過往號碼
* **使用者故事**：作為一名**運彩愛好者**，我希望能夠**查看過往中獎號碼的歷史統計數據**，以便於**分析哪些號碼出現的頻率最高或最久沒出現**。
* **主要特點**：
  * 支援導入與更新歷史中獎數據。
  * 提供號碼出現頻率（冷熱門號碼）統計圖表或列表。

### F-02：生成最可能數字組合
* **使用者故事**：作為一名**想試手氣但沒有頭緒的用戶**，我希望系統能**根據統計結果自動生成一組最有可能的中獎數字組合**，以便於**我直接進行投注**。
* **主要特點**：
  * 基於歷史數據統計結果（如熱門號碼加權或冷門號碼排除）的推薦演算法。
  * 提供一鍵生成推薦號碼的功能。

### F-03：選擇運彩種類
* **使用者故事**：作為一名**擁有多樣化投注喜好的用戶**，我希望能夠**選擇不同種類的公益彩券（如大樂透、六合彩等）**，以便於**針對特定彩券進行對應的數據分析與預測**。
* **主要特點**：
  * 支援大樂透（Lotto 6/49）、六合彩等台灣常見合法公益彩券類型。
  * 介面需提供下拉選單或切換標籤頁，切換後自動載入該彩券的規則與歷史數據。

### F-04：統計花費與盈虧記帳
* **使用者故事**：作為一名**希望進行財務管理的人**，我希望能夠**記錄每次購買彩券的花費與中獎金額**，以便於**系統幫我統計總共輸了多少錢或賺了多少錢**。
* **主要特點**：
  * 提供「新增投注紀錄」表單（包含：購買日期、彩券類型、投注金額、中獎金額）。
  * 儀表板顯示「累計花費」、「累計中獎」、「淨盈虧（淨損益）」及「投資報酬率 (ROI)」。

### F-05：使用者登入系統
* **使用者故事**：作為一名**重視隱私與個人數據的用戶**，我希望能夠**註冊並登入個人帳號**，以便於**我個人的投注記帳與分析數據不被他人看見，且能在不同裝置上同步**。
* **主要特點**：
  * 提供安全的使用者註冊、登入與登出功能。
  * 個人數據庫隔離，確保每位使用者只能看到自己的投注與記帳紀錄。

## 3. 非功能需求 (Non-functional Requirements)

### 3.1 技術限制
根據課堂規範，本專案將嚴格遵守以下技術選型：
* **前端**：純 HTML / CSS / JavaScript（不使用任何前端框架如 React/Vue/Angular）。樣式使用 Vanilla CSS 搭配 Bootstrap 5 進行快速響應式佈局。
* **後端**：Python 搭配 Flask 輕量級網頁框架。
* **模板引擎**：Jinja2（負責在後端渲染 HTML 頁面，無前後端分離）。
* **資料庫**：SQLite（透過 Python 內建 `sqlite3` 庫或 `Flask-SQLAlchemy` 進行操作）。
* **部署與版本控制**：使用 Git 進行版本控制，以 GitHub 作為協作與部署平台。

### 3.2 效能與安全性考量
* **易用性**：界面設計需直觀易懂，考量到目標用戶包含中高年齡層（如家長），字體需適度加大，操作路徑需盡量簡化。
* **安全性**：
  * 密碼安全：使用者密碼在寫入 SQLite 資料庫前必須經過雜湊加密（如使用 `werkzeug.security` 的 `generate_password_hash`）。
  * 數據隱私：嚴格限制未登入用戶存取個人記帳路由，且每筆帳目查詢需驗證當前 Session 中的 user_id，防止越權訪問（不能讓其他人知道你花多少錢）。
* **相容性**：只要使用者裝置可以連接網際網路並使用現代瀏覽器，即可順暢操作系統。

## 4. MVP 範圍 (Minimum Viable Product Scope)

為了在規定的時程內（6/15 前）順利完成並進行 Demo，我們將功能劃分為以下優先級：

### 核心功能 (Must Have) — MVP 階段
* **F-03 選擇運彩種類**：至少完整支援「大樂透」的規則與預測。
* **F-01 統計號碼**：大樂透歷史號碼的錄入與基礎頻率統計。
* **F-02 生出結果**：提供基於頻率統計的「熱門推薦」與「隨機電腦選號」功能。
* **F-04 統計花了多少錢**：基本的投注與中獎記帳表單，以及首頁的總盈虧統計。
* **F-05 登入系統**：基礎的使用者註冊、登入、登出，與 Session 狀態管理。

### 進階功能 (Should Have)
* **多種彩券支援**：除大樂透外，新增六合彩或今彩539的完整支援。
* **數據圖表視覺化**：使用 Chart.js 在前端繪製熱門號碼的直方圖或折線圖，提升視覺吸引力。
* **防過度沉迷提醒**：當累計虧損達到特定金額時，系統給予醒目的溫馨提示（落實「讓他們快點放棄賭博」的設計初衷）。

### 加分功能 (Nice to Have)
* **模擬投注器**：允許使用者用虛擬代幣模擬投注，驗證歷史勝率，不花任何真錢。
* **深色模式切換**：符合現代網頁美學，提供更護眼且質感的暗黑模式。
